# Phase 7: Settings + Export - Research

**Researched:** 2026-03-22
**Domain:** Settings CRUD UI, CSV export, Express API, React forms
**Confidence:** HIGH

## Summary

Phase 7 adds two functional areas to the existing React SPA: (1) a Settings page for managing ICP scoring rules, suppression list, daily limits, signal keywords, message templates, and viewing the cron schedule; and (2) CSV export functionality for leads with filter and date-range support.

The codebase already has all the building blocks: Express 5 API routes with JWT auth middleware (`src/api/middleware.js`), React Query hooks for data fetching (`frontend/src/hooks/useLeads.js`), Tailwind CSS 4 for styling, and Supabase tables for `icp_rules`, `watchlist`, and `suppression_list`. The settings data is currently stored across Supabase tables (`icp_rules` for scoring rules, `watchlist` for signal sources/keywords) and environment variables (`DAILY_INVITATION_LIMIT`, message templates are generated by Claude). The CSV export is a server-side concern -- generate CSV from the existing `/api/leads` query with additional date-range params and return as a downloadable file.

**Primary recommendation:** Build a new `src/api/settings.js` Express router with CRUD endpoints for `icp_rules`, `suppression_list`, and `watchlist` tables, a read-only cron schedule endpoint, and a config endpoint for daily limits. Add a `GET /api/leads/export` endpoint that streams CSV. On the frontend, add a `/settings` page with tabbed sections and an export button on the Pipeline/Sequences pages.

### Pattern 1: Settings API Router with CRUD

**What:** Single Express router (`src/api/settings.js`) with sub-routes per resource, mounted at `/api/settings`
**When to use:** For all CONF-* requirements
**Example:**
```javascript
// src/api/settings.js
const { Router } = require("express");
const authMiddleware = require("./middleware");
const { supabase } = require("../lib/supabase");

const router = Router();
router.use(authMiddleware);

// CONF-01: ICP Rules CRUD
router.get("/icp-rules", async (req, res) => {
  const { data, error } = await supabase.from("icp_rules").select("*").order("category");
  if (error) return res.status(500).json({ error: error.message });
  res.json({ rules: data });
});

router.post("/icp-rules", async (req, res) => {
  const { category, value, key, numeric_value, threshold } = req.body;
  // Validate category against allowed values
  const { data, error } = await supabase.from("icp_rules").insert({ category, value, key, numeric_value, threshold }).select().single();
  if (error) return res.status(500).json({ error: error.message });
  res.json(data);
});

router.delete("/icp-rules/:id", async (req, res) => {
  const { error } = await supabase.from("icp_rules").delete().eq("id", req.params.id);
  if (error) return res.status(500).json({ error: error.message });
  res.json({ ok: true });
});
```

### Pattern 2: CSV Export as Streaming Response

**What:** Reuse existing leads query, serialize to CSV, return as download
**When to use:** EXP-01, EXP-02, EXP-03
**Example:**
```javascript
// In src/api/leads.js -- new GET /export endpoint
router.get("/export", async (req, res) => {
  const { status, tier, source, search, date_from, date_to } = req.query;

  let query = supabase.from("leads").select("first_name, last_name, email, linkedin_url, icp_score, tier, status, created_at");

  // Apply same filters as list endpoint
  if (status) query = query.in("status", status.split(","));
  if (tier) query = query.eq("tier", tier);
  if (source) query = query.eq("signal_category", source);
  if (date_from) query = query.gte("created_at", date_from);
  if (date_to) query = query.lte("created_at", date_to);

  const { data, error } = await query.order("created_at", { ascending: false });
  if (error) return res.status(500).json({ error: error.message });

  // Build CSV
  const headers = ["Nom", "Prenom", "Email", "LinkedIn", "Score ICP", "Tier", "Statut", "Date"];
  const rows = data.map(l => [
    l.last_name || "", l.first_name || "", l.email || "", l.linkedin_url || "",
    l.icp_score ?? "", l.tier || "", l.status || "", l.created_at || ""
  ]);

  const csv = [headers, ...rows].map(r => r.map(v => '"' + String(v).replace(/"/g, '""') + '"').join(",")).join("\n");

  res.setHeader("Content-Type", "text/csv; charset=utf-8");
  res.setHeader("Content-Disposition", 'attachment; filename="leads-export.csv"');
  // BOM for Excel UTF-8 compatibility
  res.send("\uFEFF" + csv);
});
```

### Anti-Patterns to Avoid
- **Editing .env from UI:** Never write to .env files from the web interface. Store mutable config in Supabase `settings` table.
- **Exposing hashed values:** Suppression list stores SHA-256 hashes for RGPD. Never display raw emails. Show count + source only, allow adding new entries that get hashed server-side.
- **CSV in browser:** Building CSV client-side limits to data already fetched. Server-side export can handle the full dataset without pagination limits.
- **Restarting PM2 from UI:** Cron schedule is read-only for a reason (noted in REQUIREMENTS Out of Scope). Never trigger PM2 restart from the web interface.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| CSV generation with special chars | Full CSV parser with streaming | Simple `map + join` with proper quoting and BOM | Data is flat, 8 columns, no nested objects. Hand-rolled is adequate here. |
| Tab component library | Custom accessible tab system | Simple `useState` + conditional render | Single user app, 6 tabs, no accessibility concerns for solo use |
| Form validation library | Zod/Yup schema validation | Inline validation in API + basic client checks | Single user, simple forms, server-side validation is sufficient |

**Key insight:** This is a single-user admin panel. Simplicity wins over robustness. Server-side validation protects data integrity; client-side is nice-to-have for UX.

## Common Pitfalls

### Pitfall 1: Breaking ICP Scorer by Deleting All Rules
**What goes wrong:** User deletes all ICP rules, next task-a run scores everything as 0/cold.
**Why it happens:** `loadIcpRules()` returns empty array, `buildScoringPrompt()` generates a prompt with all "non specifie" values.
**How to avoid:** Warn in UI before deleting last rule in a category. Consider minimum rule set validation on the API side.
**Warning signs:** All new leads scored cold after settings change.

### Pitfall 2: CSV Export Timeout on Large Datasets
**What goes wrong:** If leads table grows to thousands, SELECT * with no pagination may time out.
**Why it happens:** Supabase free tier has query time limits.
**How to avoid:** Limit export to 10,000 rows max. Add `.limit(10000)` to export query. Show count to user before export.
**Warning signs:** 500 error on export with large datasets.

### Pitfall 3: Suppression List Hash Collisions with Manual Entry
**What goes wrong:** User adds a raw email/LinkedIn URL to suppression list; it needs to be SHA-256 hashed server-side to match the existing `hashed_value` column format.
**Why it happens:** Existing code in `leads.js` already hashes with `crypto.createHash("sha256")`. Manual entries must use the same hashing.
**How to avoid:** Always hash on the server side in the API endpoint, never on the client. Reuse the existing hashing pattern from `leads.js`.
**Warning signs:** Manual suppression entries not being matched during dedup.

### Pitfall 4: Message Templates Decoupled from Generator
**What goes wrong:** User edits template text but message-generator.js still uses hardcoded prompts.
**Why it happens:** `message-generator.js` has prompts inline, not loaded from DB.
**How to avoid:** Modify `message-generator.js` to load template text from the `settings` table (with fallback to current hardcoded values). Load once per task run (like `loadIcpRules()`).
**Warning signs:** Template changes in UI have no effect on generated messages.

### Pitfall 5: Watchlist Edit Breaks Running Task
**What goes wrong:** User edits watchlist while task-a is running; task may see inconsistent state.
**Why it happens:** task-a loads watchlist once at start (`collectSignals()`), so mid-run edits are safe for the current run. Next run picks up changes.
**How to avoid:** This is actually fine by design -- no action needed. Document that changes take effect on next task-a run.
**Warning signs:** None -- this is a non-issue.

### CSV Export with UTF-8 BOM for Excel
```javascript
// Excel requires BOM (\uFEFF) to correctly interpret UTF-8 CSV with accented characters
function buildCsv(headers, rows) {
  const escape = (v) => '"' + String(v).replace(/"/g, '""') + '"';
  const lines = [headers.map(escape).join(",")];
  for (const row of rows) {
    lines.push(row.map(escape).join(","));
  }
  return "\uFEFF" + lines.join("\n");
}
```

## Open Questions

1. **Message template granularity**
   - What we know: `message-generator.js` has 5 functions with distinct hardcoded prompts (invitation, followup, email, whatsapp, inmail)
   - What's unclear: Should the user edit the full system prompt or just key instructions/rules within each template?
   - Recommendation: Store the user-facing "rules" section as editable text. Keep the system prompt and JSON output format fixed. This gives control without risking broken JSON output.

2. **ICP rules table schema precision**
   - What we know: `icp-scorer.js` reads categories and uses `value`, `key`, `numeric_value`, `threshold` fields
   - What's unclear: Exact Supabase schema (column types, constraints) -- inferred from code only
   - Recommendation: During planning, the first task should query the actual Supabase schema via the API to confirm column names and types before building the CRUD UI.

3. **Daily limits: env var vs DB**
   - What we know: `DAILY_INVITATION_LIMIT` is read from env in task-b; daily lead cap of 50 is hardcoded in task-a
   - What's unclear: Should the settings table override env vars, or replace them entirely?
   - Recommendation: Settings table takes priority; fall back to env var; fall back to hardcoded default. Pattern: `settings.daily_invitation_limit || process.env.DAILY_INVITATION_LIMIT || 15`.
