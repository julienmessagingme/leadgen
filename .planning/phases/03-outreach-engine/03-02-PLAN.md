  <action>
Replace the placeholder task-b-invitations.js on VPS with full implementation. Module exports `async function taskBInvitations(runId)`.

**Flow:**
1. Log task start.
2. **LIN-05:** Call `checkLimits()` from bereach.js. Log the limits response. If BeReach reports limit exceeded, log and return early.
3. **LIN-03 double check:** Read `DAILY_INVITATION_LIMIT` from env (default 15). Query Supabase leads table: count rows where `invitation_sent_at >= todayStartParis()`. Calculate `remaining = limit - todaySent`. If remaining <= 0, log "daily limit reached" and return.
4. **Select leads:** Query Supabase `leads` table: status IN ('new', 'enriched', 'scored') -- whichever statuses indicate the lead is ready but not yet contacted. Tier IN ('hot', 'warm'). Order by icp_score DESC. Limit to `remaining`.
5. **For each lead:**
   a. **LIN-08 idempotence:** Query Supabase logs: if entry exists with run_id=current AND lead_id=lead.id AND task='task-b-invitations' AND level='info' AND message contains 'Invitation sent' -- skip this lead.
   b. **Suppression check:** Call `isSuppressed(lead.email, lead.linkedin_url)`. If suppressed, log and continue.
   c. **LIN-02:** Call `generateInvitationNote(lead)` from message-generator.js. If null (generation failed), log warning and skip lead.
   d. **LIN-01:** Call `connectProfile(lead.linkedin_url, note)` from bereach.js.
   e. **Update lead:** Set status='invitation_sent', invitation_sent_at=now(), store note and run_id in metadata JSONB field.
   f. **Log:** Log success with lead name and note length.
   g. **LIN-04:** If not last lead, sleep `60000 + Math.floor(Math.random() * 60000)` ms (60-120s) using sleep() from bereach.js.
6. Wrap each lead iteration in try/catch for error isolation (one lead failure does not crash the batch).
7. Log summary: X invitations sent, Y skipped, Z errors.

**Helper:** Reuse `getTodayStartParis()` from task-a-signals.js -- either extract to a shared util or copy the function. Prefer copying to avoid modifying task-a.

**Imports:** supabase, checkLimits, connectProfile, sleep (from bereach), isSuppressed (from suppression), generateInvitationNote (from message-generator), log (from logger).

  <name>Task 2: Implement Task C -- LinkedIn connection follow-up messages</name>

  <action>
Replace the placeholder task-c-followup.js on VPS with full implementation. Module exports `async function taskCFollowup(runId)`.

**Flow:**
1. Log task start.
2. **LIN-06 -- Detect accepted connections:**
   a. Call `getSentInvitations()` from bereach.js -- returns list of PENDING sent invitations.
   b. Query Supabase leads with status='invitation_sent' (these are leads we invited).
   c. For each lead with status='invitation_sent': if their linkedin_url is NOT in the pending invitations list, their invitation was accepted (or withdrawn -- we treat both as "connected" since we cannot distinguish).
   d. Update these leads' status to 'connected'.
3. **LIN-07 -- Send follow-up messages to newly connected leads:**
   a. Query Supabase leads with status='connected' AND metadata->follow_up_sent IS NULL (or follow_up_sent_at IS NULL on the lead).
   b. For each connected lead:
      - Check idempotence (LIN-08): skip if already processed in this run.
      - Check suppression list.
      - Call `generateFollowUpMessage(lead)` from message-generator.js. If null, skip.
      - Call `sendMessage(lead.linkedin_url, message)` from bereach.js.
      - Update lead: set follow_up_sent_at=now(), store message in metadata.
      - Log success.
      - Sleep 60-120s between messages (same rate limiting as Task B).
4. Error isolation per lead in try/catch.
5. Log summary.

**Note on connection detection:** The BeReach `/invitations/linkedin/sent` returns pending invitations. If a lead's invitation is no longer pending, we assume accepted. This is an approximation -- some may have been withdrawn or expired. This is acceptable per the research open questions.

**Imports:** supabase, getSentInvitations, sendMessage, sleep (from bereach), isSuppressed (from suppression), generateFollowUpMessage (from message-generator), log (from logger).
